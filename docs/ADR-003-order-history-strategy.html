Architecture Decision Record

ADR-003: Order History Migration Strategy

Accepted

Context

When migrating from WooCommerce to BigCommerce, historical order data presents a significant decision point:

  1. Customer Expectations: Customers expect to see their order history in their account
  2. Business Operations: Staff need access to historical orders for customer service
  3. Analytics Continuity: Historical data enables trend analysis and forecasting
  4. Migration Complexity: Order data includes complex relationships (customers, products, addresses, payment records)
  5. Legal/Compliance: Some industries require order record retention

Data Considerations

WooCommerce stores orders with:

  • Order metadata in wp_postmeta (EAV pattern)
  • Line items with product references
  • Customer addresses (billing/shipping)
  • Payment transaction records
  • Order notes and status history
  • Refund records

BigCommerce order structure differs significantly, requiring transformation.

Decision

We will implement a tiered order history migration strategy.

Tier 1: Recent Orders

< 12 months
  • Full migration to BigCommerce via API
  • Complete order details including line items, addresses, notes
  • Enables customer self-service

Tier 2: Historical Orders

12-36 months
  • Archive export to structured JSON/CSV
  • Import to BigCommerce as read-only (where supported)
  • Or maintain WordPress archive access

Tier 3: Legacy Orders

> 36 months
  • Export to archival format (JSON, CSV, PDF)
  • Do not migrate to BigCommerce
  • Retain in separate archive system

Rationale

Why Not Migrate All Orders?

  1. API Limits: BigCommerce API has rate limits; bulk historical import is slow
  2. Diminishing Returns: Older orders are rarely accessed
  3. Data Quality: Legacy WooCommerce orders often have inconsistent data
  4. Product References: Historical products may not exist in new catalog
  5. Cost/Time: Full migration adds weeks to timeline with minimal benefit

Why Not Skip Order Migration?

  1. Customer Experience: Recent orders are actively referenced
  2. Support Operations: Customer service needs recent order access
  3. Subscription Continuity: Subscription customers need order history
  4. Returns/Exchanges: Recent orders may have active return windows

Customer Communication

Pre-migration communication should set expectations:

  • "Your recent order history will be available in your new account"
  • "For orders older than [date], please contact support"
  • Provide PDF export option for customers who want full history

Consequences

Positive

  • Faster Migration: Reduced data volume speeds timeline
  • Cleaner Data: Only recent, higher-quality data migrates
  • Customer Focus: Prioritizes orders customers actually need
  • Compliance Ready: Archive strategy maintains required records

Negative

  • Split History: Customers may need to access two systems
  • Staff Training: Support staff need to know where to find historical orders
  • Archive Maintenance: Must maintain archive system long-term

Alternatives Considered

Alternative 1: Migrate All Orders

Full historical migration regardless of age.

Rejected: Excessive time/cost, diminishing returns, data quality issues

Alternative 2: Migrate No Orders

Clean slate approach.

Rejected: Poor customer experience, support burden, subscription issues

Alternative 3: Customer Self-Service Export

Let customers export their own history before migration.

Rejected: Low adoption rate, still need support fallback
