ADR-002: Headless Commerce vs Checkout-Only

Accepted

Context

When integrating BigCommerce with WordPress, there are two primary architectural approaches:

Option A: Checkout-Only (Buy Button)

  • WordPress displays products (from WooCommerce or custom)
  • BigCommerce only handles checkout and payment processing
  • Minimal integration surface

Option B: Full Headless Commerce

  • BigCommerce becomes the source of truth for all product data
  • Products sync to WordPress as custom post types
  • BigCommerce handles checkout, inventory, orders, and fulfillment
  • WordPress handles presentation, content, and SEO

The decision impacts: migration complexity, ongoing maintenance burden, feature availability, performance characteristics, and data consistency.

Decision

We will implement full headless commerce architecture using the BigCommerce for WordPress plugin.

Products, categories, and commerce data live in BigCommerce and sync to WordPress for display. WordPress remains the presentation layer while BigCommerce handles all commerce operations.

Rationale

Why Not Checkout-Only?

  1. Data Duplication: Products would need to exist in both WooCommerce and BigCommerce
  2. Sync Complexity: Two sources of truth creates reconciliation challenges
  3. Feature Loss: Can't leverage BigCommerce's full feature set (B2B, multi-channel, etc.)
  4. Incomplete Migration: Doesn't address WooCommerce maintenance burden—still running two commerce platforms

Why Full Headless?

  1. Single Source of Truth: BigCommerce is the canonical product database
  2. Full Feature Access: B2B pricing, multi-storefront, inventory management
  3. Clean Migration: WooCommerce can be fully decommissioned after migration
  4. Scalability: BigCommerce handles commerce scale; WordPress handles content
  5. SEO Friendly: Products become WordPress custom post types—fully indexable

Consequences

Positive

  • Clean Architecture: Clear separation of concerns
  • Scalable Commerce: 1000s of concurrent checkouts without impacting WordPress
  • PCI Compliance: Hosted checkout keeps WordPress outside PCI scope
  • Feature Complete: Access to all BigCommerce features including B2B Edition
  • API-First: Strong APIs (20,000 requests/hour) enable custom integrations

Negative

  • Sync Latency: Product changes take time to reflect in WordPress
  • Plugin Dependency: Relies on BC Bridge plugin
  • Checkout Redirect: Users leave WordPress domain for checkout

Alternatives Considered

Alternative 1: Embed Checkout Widget

Embed BigCommerce checkout in WordPress iframe.

Rejected: Poor UX, mobile issues, limited customization

Alternative 2: Hybrid with WooCommerce Retained

Keep WooCommerce for simple products, BigCommerce for complex.

Rejected: Maintains two commerce platforms, doubles complexity

Alternative 3: Custom Headless Implementation

Build custom React/Vue frontend with BC Storefront API.

Rejected: Higher development cost, loses WordPress content benefits
