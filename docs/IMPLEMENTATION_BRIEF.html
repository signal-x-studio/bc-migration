BC Migration: Implementation Brief

For technical stakeholders who've read Granite/Artemis and want to understand the implementation approach

The Approach in One Sentence

Keep WordPress as your content layer, move only the commerce engine to BigCommerce. Our toolkit assesses WooCommerce stores, migrates commerce data via API, and connects the platforms through the BC Bridge plugin—preserving SEO, content, and team workflows.

Key Differentiators

No Content Migration

Blog posts, pages, media, themes—all stay in WordPress. Only commerce data moves.

SEO Preserved

WordPress URLs unchanged. WooCommerce's 92% SEO score stays intact.

Team Continuity

Marketing keeps WordPress. Only commerce ops learns BigCommerce admin.

Getting Started

# Clone and configure git clone <repo> && cd bc-migration npm install cp .env.example .env # Add WooCommerce + BigCommerce API keys # Run assessment (non-destructive) npm run assess # Review readiness report, then migrate npm run migrate # Validate results npm run validate

The assessment is read-only and generates a readiness report with complexity scoring, plugin analysis, and migration recommendations.

What About WordPress Catalyst?

If you're building a headless BigCommerce storefront (Catalyst, Stencil, or Makeswift) but want to pull WordPress content into it, we have React components for that. The WordPress Catalyst Components render blog posts, pages, and navigation from the WordPress REST API with theme-aware styling.

This is the inverse of BC Bridge—instead of putting BigCommerce inside WordPress, you're putting WordPress content inside a BigCommerce-powered frontend.
