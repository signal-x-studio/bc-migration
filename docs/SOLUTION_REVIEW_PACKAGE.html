WooCommerce Migration Solution: Review Package

Status: Ready for Review | Date: December 2025

Executive Summary

The Problem

WooCommerce merchants want to upgrade to BigCommerce but face two barriers:

  1. Migration complexity - Data migration is manual, error-prone, and expensive
  2. WordPress lock-in - They don't want to abandon their WordPress content, SEO equity, and editorial workflows

The Solution

A migration toolkit + WordPress integration that lets merchants:

  • Keep WordPress (content, blog, theme, SEO)
  • Move commerce operations to BigCommerce
  • Migrate data automatically (products, categories, customers)
  • Minimize downtime and data loss
Key Value Proposition: "Keep WordPress. Upgrade Commerce."

Decision 1: Order History Migration

Question: Should we migrate historical orders to BigCommerce?

Option A: Don't migrate Recommended

Pros: Simple, fast, no data loss risk

Cons: Split admin (WP for old, BC for new)

Rationale: Most merchants accept split admin for historical data. This is the simplest path for MVP.

Option B: Import as read-only Consider for Phase 2

Pros: Unified view in BC

Cons: BC import limitations, risk of data issues

Option C: Unified dashboard Future Enhancement

Pros: Best UX

Cons: Custom development required

Input needed from: Commerce.com Product, WPEngine customers

Decision 2: BC Bridge Plugin Ownership

Question: Who builds and maintains the WordPress plugin?

Option A: WPEngine builds WPEngine

Pros: WordPress expertise, WPEngine integration

Cons: Resource allocation, roadmap alignment

Option B: Commerce.com builds Commerce.com

Pros: Control, integration with migration tool

Cons: WordPress expertise gap

Option C: Partner/agency builds External

Pros: External resource

Cons: Coordination overhead, IP ownership

Option D: Open source community Community

Pros: Ecosystem, contributions

Cons: Quality control, support burden

Input needed from: WPEngine leadership, Commerce.com leadership

Decision 3: Products Exceeding 600 Variants

Question: How do we handle WC products with more than 600 variants?

Option A: Split into multiple BC products Recommended

Pros: Works within limits

Cons: Customer UX change, complexity

Option B: Use BC modifiers instead of variants

Pros: Avoids limit

Cons: Different purchasing UX

Option C: Block migration, require manual restructure Fallback

Pros: Clear, simple

Cons: Poor merchant experience

Option D: Negotiate BC limit increase

Pros: Best UX

Cons: May not be possible

Input needed from: BC Platform team (is limit increase possible?)

Decision 4: Checkout Experience

Question: Where does checkout happen?

Option A: Redirect to BC hosted checkout Recommended for MVP

Pros: Simplest, PCI handled

Cons: UX transition, domain change visible

Option B: BC Embedded Checkout in WP Future Enhancement

Pros: Seamless UX

Cons: More integration work, iframe limitations

Option C: Custom checkout via BC API

Pros: Full control

Cons: Highest complexity, PCI scope

Input needed from: Commerce.com Product, WPEngine (any checkout partnerships?)

Decision 5: Feedonomics Role

Question: Should Feedonomics be the ETL layer for migrations?

Option A: Yes, use Feedonomics

Pros: Existing asset, proven at scale

Cons: Adds dependency, may be overkill for migration

Option B: No, build custom ETL

Pros: Tailored to migration, simpler

Cons: Duplicate capability

Option C: Hybrid Likely Recommendation

Pros: Feedonomics for ongoing sync, custom for initial migration

Cons: Complexity

Input needed from: Feedonomics team - need to understand capabilities first
